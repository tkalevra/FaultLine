# Entity ID Normalization

## Problem

FaultLine stores facts with entity references (subject_id, object_id). These must be **UUID v5 surrogates** generated by `EntityRegistry._make_surrogate()`. However, due to legacy code or edge cases, some facts may have been stored with string entity_ids (e.g., "marla", "fraggle") instead of UUIDs.

String entity_ids break:
- Graph traversal in `/query` endpoint (assumes UUIDs for reliable lookups)
- Entity type tracking in `entities` table
- Fact deduplication and conflict detection

## Migration

The system includes automated detection and a standalone migration script.

### Automatic Detection (Startup)

The `/ingest` endpoint now:
1. **Validates** that all entity_ids are UUIDs before storing (rejects string IDs with warning log)
2. **Checks** at startup for existing string entity_ids and logs a warning

If you see this log at startup:
```
startup.entity_id_denormalization_detected facts_with_string_ids=5 staged_facts_with_string_ids=2
```

You need to run the migration.

### Manual Normalization

Run the standalone migration script:

```bash
cd /path/to/FaultLine
export POSTGRES_DSN="postgresql://user:pass@localhost:5432/faultline"
python3 migrations/018_normalize_entity_ids.py
```

The script:
1. **Scans** facts and staged_facts for string entity_ids
2. **Generates** proper UUID v5 surrogates using the same logic as `EntityRegistry._make_surrogate()`
3. **Updates** all references atomically
4. **Syncs** entity_aliases to preserve display names
5. **Verifies** success

### Safety

The migration is **idempotent** — running it multiple times is safe:
- UUID entity_ids are skipped (only strings are normalized)
- Existing aliases are preserved
- All updates are atomic (full rollback on error)

### Manual Verification

After running the migration, verify in the database:

```sql
-- Check for remaining string entity_ids
SELECT COUNT(*) FROM facts
WHERE subject_id NOT LIKE '%-%-%-%-' OR object_id NOT LIKE '%-%-%-%-';

SELECT COUNT(*) FROM staged_facts
WHERE subject_id NOT LIKE '%-%-%-%-' OR object_id NOT LIKE '%-%-%-%-';

-- Both should return 0
```

## Internals

### EntityRegistry._make_surrogate()

```python
def _make_surrogate(user_id: str, name: str) -> str:
    """Generate deterministic UUID v5 surrogate.
    
    For user_id='anonymous' and name='marla':
    1. Create namespace: uuid5(_FAULTLINE_NAMESPACE, 'anonymous')
    2. Generate surrogate: uuid5(namespace, 'marla')
    Result: Deterministic, repeatable UUID
    """
```

### String ID Detection

UUIDs match pattern: `^[0-9a-f]{8}-[0-9a-f]{4}-...$`

Strings like "marla", "fraggle", "chris" don't match this pattern.

Database filter: `WHERE entity_id NOT LIKE '%-%-%-%-'`
