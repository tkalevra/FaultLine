# A note on how FaultLine has been described


FaultLine has been in development for a while(two months, solo dev), and some of the documentation written
along the way — including prod-facing docs and earlier versions of the architecture
notes — described it as a **vector-mirrored, RAG-style system**: facts flowing into
a vector index that the LLM then retrieves by similarity. A few diagrams put Qdrant
front and center as if it were the memory.

That description was wrong, and I want to correct it plainly rather than quietly
edit it away.

The summary below has been generated by AI, but I'm here and actively reviewing output and validation against claims!

## The honest version

- **PostgreSQL is the authoritative store.** The knowledge graph in PostgreSQL —
  Class A (user-stated) and Class B (inferred-but-grounded) facts — is the
  long-term memory. Every answer the system serves resolves to a real row.
- **The vector index (Qdrant) is a short-term scratchpad, not the source of
  truth.** It holds **Class C only** — the "couldn't classify this yet" tier — and
  acts as a backstop and a promotion gate. When a Class C item proves itself, it is
  promoted into the structured graph (C→B); otherwise it expires. Class A/B are
  never vector-indexed and never served from the vector lane.
- **Retrieval is a deterministic walk, not a similarity search.** The query path
  resolves an anchor and walks the graph. It does not rank text by cosine distance
  and hand chunks to the model to interpret. Same question, same walk, same rows.
- **The vector never overrides PostgreSQL.** It surfaces short-term material and
  helps promote it; the structured store is always authoritative.

This is the opposite of the usual layout, where a vector store is the long-term RAG
library. FaultLine inverts it: **the graph is long-term and authoritative; the
vector is short-term and derived.**

## Why the old framing happened

"Validated memory for an LLM, backed by Postgres and Qdrant" reads a lot like RAG
from the outside, and early notes leaned on that shorthand because it was familiar.
As the deterministic walk became the real retrieval path and the vector got demoted
to the Class-C tier, the docs didn't keep up. The code moved; some of the prose
didn't.

TL;DR I wasn't watching the needle closely enough and the system started treating vector as truth, which isn't right.

We'd(I love that claude includes itself here) rather be precise about what this is — a deterministic, write-validated
knowledge-graph memory — than let a convenient but inaccurate RAG label stand. If
you find a place in the docs that still implies vector-primary or RAG-style
behavior, that's a bug in the docs; please open an issue.

For the mechanics, see [docs/ARCHITECTURE.md](docs/ARCHITECTURE.md), especially the
**"Why this isn't RAG (mechanically)"** section.
