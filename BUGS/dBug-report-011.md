# dBug-011: Incomplete Bidirectional Relationships (parent_of/child_of)

**Status**: Open  
**Severity**: Medium  
**Reported**: 2026-05-12  
**Environment**: Pre-prod (192.168.40.10:8001)  
**Root Cause**: Extraction completeness + ingest gap — LLM extracts only one direction of bidirectional relationships  

## Summary

**CONFIRMED (from pre-prod logs 2026-05-12 02:31:58–02:32:14):**

`/query` returns 46–47 facts successfully. Filter receives facts from API correctly. However, **bidirectional relationships are incomplete** — only one direction of parent/child relationships exists.

Evidence:
- Filter shows: `gabby -child_of-> user` ✓
- Filter shows: NO `user -parent_of-> gabby` ✗
- Filter's family summary lists only "children=Des, Cyrus" (missing Gabby)
- Gabby's age IS present: `gabby -age-> 10` ✓ (confirming partial data retrieval)

**NOT a query failure. NOT a database failure.** The bug is extraction incompleteness + ingest gap.

## Root Cause

**Bidirectional relationship gap in extraction + ingest pipeline:**

1. **LLM Extraction** (`openwebui/faultline_tool.py`, line 86–171)
   - `_TRIPLE_SYSTEM_PROMPT` defines parent_of/child_of semantics (lines 96–99)
   - BUT does NOT instruct LLM to emit **both directions** for bidirectional rels
   - When text says "I have children: Des, Cyrus, Gabby," LLM extracts:
     ```
     user -parent_of-> des
     user -parent_of-> cyrus
     user -parent_of-> gabby  ← extraction incomplete or fails here
     ```
   - The inverse direction (`gabby -child_of-> user`) is NOT generated by LLM

2. **Ingest Gap** (`src/api/main.py`, line ~2590)
   - `_validate_bidirectional_relationships()` currently:
     - ✓ Detects conflicts (both directions exist with different confidence)
     - ✓ Keeps higher confidence, supersedes lower
     - ✗ Does NOT create missing inverses
   - No function exists to auto-generate inverse relationships from one-directional input

**Result:** Only incomplete unidirectional facts are stored. Filter later retrieves `gabby -child_of-> user` (from some source, possibly graph traversal), but the canonical `user -parent_of-> gabby` never existed.

## Expected Behavior

- Both directions of bidirectional relationships should exist: `(A, parent_of, B)` AND `(B, child_of, A)`
- Filter should receive complete family facts for all children
- Family summary should include all children: "spouse=Mars, children=Des, Cyrus, Gabby"

## Actual Behavior

- Only one direction is extracted/stored (bidirectional completeness gap)
- Filter receives partial facts and produces incomplete family summaries
- `/query` correctly returns what exists; the problem is upstream in extraction/ingest

**Step 2: Review LLM extraction prompt**
- File: `openwebui/faultline_tool.py`
- Section: `_TRIPLE_SYSTEM_PROMPT` (lines 86–171)
- Check: Does it instruct LLM to emit both directions of bidirectional relationships?
- Current state (line 96–99): defines semantics but does NOT instruct bidirectional emission

**Step 3: Fix extraction prompt**
Add guidance like:
```
BIDIRECTIONAL EMISSION:
For relationships with inverses (parent_of↔child_of, spouse↔spouse, sibling_of↔sibling_of),
emit BOTH directions as separate facts.
Example: "I have a son named Des" 
→ emit (user, parent_of, des) AND (des, child_of, user)
```

**Step 4: Add ingest-level safety net**
Update `_validate_bidirectional_relationships()` in `src/api/main.py` to also:
- Create missing inverse relationships when only one direction exists
- Example: If only `(A, parent_of, B)` arrives, auto-create `(B, child_of, A)`
- Prevents future extraction gaps from causing data loss
